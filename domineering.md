### 1. 시작

> **2026. 05. 03. ~ 05. 04.**

처음 봇을 구상할 때는 욕심이 많았다. 머신러닝용 selfplay 파이프라인도 구축해보고, nnue를 써볼 생각을 이때부터 하고있었다.

하지만 생각보다 전체 10초라는 시간관리가 너무 빡셌다. 꽤 정성을 들인 NN+MCTS보다, GPT 5.5가 한번에 만들어낸 alphabeta 기반의 봇이 훨씬 강했다. 결국 고민끝에 무거운 파이프라인을 유지하는 것보다 **빠르고 안정적인 엔진을 극한까지 깎는 것**이 낫다고 판단했다. (nypc codebattle때 수작업으로 튜닝한사례를 봤고, 그게 감명깊게 남은게 크다.)

이후 눈물을 머금고 거대한 ML 파이프라인을 싹 다 날려버렸다.

### 2. 개발 시간의 대부분은 어디로?

방향은 정했지만 가장 큰 문제가 남아있었다. **나는 이런 류의 휴리스틱을 극한까지 깎아본 경험이 없었다.** 직관에 의존해서 파라미터를 튜닝하다가는 망할게 뻔했다.

그래서 봇의 성능을 정량적으로 평가해 줄 **모델 구축**에 내 개발 시간의 대부분을 쏟아부었다. 
방식은 무식하지만 확실했다. GPT에게 간단한 minimax alphabeta 봇을 짜달라고 한 뒤, 그 봇들끼리 매 게임 3분씩, 수천 판을 돌려 데이터를 뽑아냈다. (지인에게 64코어 머신을 잠깐 빌렸다) 그리고 현재 보드 상태의 승률과 기대 가치를 예측하는 **ResNet 베이스의 EV model**을 학습시켰다. 꽤 그럴듯하게 EV를 계산해냈다.

### 3. 튜닝: GPT-Codex라는 치트키

꽤 정확한 EV 모델이 생기니 이후는 별로 할게 없었다.
참고로 이후에 사용할 엔진의 뼈대(전치테이블, PVS, LMR 등등)는 잘 알려진 엔진들이나 논문에서 가져다 썼다.

세부적인 하이퍼파라미터들을 내가 건드리는 대신, **GPT-Codex에 EV 모델을 붙여 자동튜닝**하도록 돌려버렸다. 그렇게 기계가 깎아낸 핵심 로직들은 꽤나 흥미로웠다.

### 4. 기계가 깎아낸 휴리스틱들

튜닝을 거치며 완성된 메인 로직 중 인상 깊었던 부분은 다음과 같다.

**1. Leaf Node Evaluation**
단순한 점수 차이뿐만 아니라 미래의 잠재력까지 철저히 수치화했다.
```cpp
e = x.p[H] - x.p[V]                  // 확정 점수 차이
  + (x.e[H] - x.e[V]) / 18           // DP 기반 잠재 점수 (타일 중첩 고려 1/18 스케일링)
  + (x.n[H] - x.n[V]) / 2            // 기동성 (내가 놓을 수 있는 남은 경우의 수)
  + (x.b[H] - x.b[V]) / 4;           // 단일 턴 최대 고득점 위협
```

**2. Move Ordering Priority**
탐색 우선순위를 정할 때, 기계는 **점수를 내는 것만큼이나 상대를 말려 죽이는 것을 중요하게 여겼다.**
```cpp
int v = m.sc * 3000                  // 이 수로 얻는 점수
      + t.ol * 2200                  // 상대의 잠재 점수 파괴 (Opponent Loss)
      - pos(t.sl) * 900              // 내가 포기하는 잠재 점수 (Self Loss)
...
v += (x.n[H] + x.n[V] - t.n[s] - t.n[s ^ 1]) * 15 
     - t.n[s ^ 1] * 35               // 상대 기동성 제어 (상대의 경우의 수를 없앰)
```
상대 잠재력 파괴의 가중치가 꽤 높았다(여기서 더 높이면 오히려 성능이 떨어진다. 안 좋은 패스를 유도하는 늪 같은 플레이가 완성된 배경이다. 실제로 로컬에서 테스트했을때 상대봇의 패스 허용 여부에 따라 승률이 차이가 난다.
